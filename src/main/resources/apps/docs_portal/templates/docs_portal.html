Enterprise Framework SDK Documentation

Welcome to the official documentation for the Enterprise Framework SDK, a lightweight, high-performance HTTP framework designed for building secure and scalable microservices.

1. Overview

The Enterprise Framework SDK is a minimalist web framework built on top of Java's native com.sun.net.httpserver.HttpServer. It is designed to provide developers with direct control over HTTP requests and responses while offering clean abstractions for routing and request parsing.

2. Core Architecture

The framework revolves around three main components:

  • Router: Dispatches incoming requests to the registered controllers based on HTTP method and path.
  • Request: Encapsulates all incoming data, including headers, query parameters, and body content.
  • Response: Allows controllers to construct HTTP responses with custom status codes, headers, and body content.

An application built with this framework typically consists of three main files:

  • Main: The entry point of the application, responsible for setting up the router, registering routes, applying global and route-specific middlewares, and starting the server.
  • Controller: Contains the handler functions that process requests and return responses.
  • Middleware: Contains filters that intercept requests and responses to perform cross-cutting concerns like authentication, logging, or security hardening.

3. Request Lifecycle

  1. A client sends an HTTP request.
  2. The server receives the request and parses the URL path and query string.
  3. The Router matches the method and path to a registered handler.
  4. A Request object is instantiated and populated with headers, query parameters, and body.
  5. The handler is executed with the Request object as its argument.
  6. The handler returns a Response object (or a dict/str which is automatically converted).
  7. The server sends the response back to the client.

4. API Reference

4.1. Request Class

Represents the incoming HTTP request. In Java, properties are accessed via getter methods.

Getter Methods

  • getMethod() (String): Returns the HTTP method (e.g., "GET", "POST") in uppercase.
  • getPath() (String): Returns the request path (e.g., "/profile").
  • getHeaders() (Map<String, String>): Returns a map of incoming HTTP headers.
  • getQueryParams() (Map<String, String>): Returns parsed query parameters.
  • getBody() (String): Returns the raw body content. Useful for reading POST JSON payloads.
  • getCookies() (Map<String, String>): Returns cookies parsed from the Cookie header.

Helper Methods

  • getHeader(String name) / getHeader(String name, String defaultValue): Case-insensitively looks up a header value.

Example

public static Response myHandler(Request request) {
    // Case-insensitive header lookup
    String userAgent = request.getHeader("User-Agent");
    
    // Accessing cookies
    String sessionId = request.getCookies().get("session_id");
    
    // Accessing raw body and parsing JSON (using framework Utils)
    Map<String, String> payload = Utils.parseJson(request.getBody());
    String username = payload.get("username");
    // ...
    return new Response("OK");
}

4.2. Response Class

Represents the outgoing HTTP response.

Constructors

new Response(String body)
new Response(String body, int statusCode)
new Response(String body, String contentType)
new Response(String body, int statusCode, String contentType)

Methods

  • addHeader(String name, String value): Adds a custom header to the response. If the header already exists, it appends the value to support multi-valued headers (e.g., Set-Cookie).
  • enableContentSecurityPolicy(): Configures a strict Content Security Policy (CSP) on the response header to prevent resources loading from untrusted sources. Note that this is not enabled by default, and enabling it may break some application functionalities.
  • disableCaching(): Prevents browsers from caching the response by setting Cache-Control, Pragma, and Expires headers to disable cache.
  • enableHsts(): Enables HTTP Strict Transport Security (HSTS) by injecting Strict-Transport-Security header, forcing HTTPS communication.
  • enableXssProtection(): Enables legacy X-XSS-Protection: 1; mode=block header for compatibility with older browsers' built-in XSS filters.
  • setReferrerPolicy(String policy): Sets the Referrer-Policy header (e.g. "strict-origin-when-cross-origin") to control how much referrer info is sent.
  • enableCors() / enableCors(String allowOrigin): Enables Cross-Origin Resource Sharing (CORS) by setting Access-Control-Allow-Origin, Access-Control-Allow-Methods, and Access-Control-Allow-Headers.
  • sanitize(String placeholder, String value): HTML-escapes the value and replaces occurrences of {{placeholder}} in the response body.

Example

public static Response myHandler(Request request) {
    Response response = new Response("Welcome!");
    response.addHeader("X-Custom-Header", "Value");
    response.enableCors("*");
    response.disableCaching();
    return response;
}

4.3. Router Class

Manages routing table, handler dispatching, and middleware execution.

Methods

  • use(Middleware middleware): Registers a global middleware that runs for all routes.
  • addRoute(String method, String path, Handler handler): Registers a handler for a specific HTTP method and path.
  • addRoute(String method, String path, Handler handler, List<Middleware> middlewares): Registers a handler with route-specific middlewares.

Middleware Specification

Middlewares are implementations of the functional interface Middleware. They allow intercepting requests before they reach the main controller handler, or modifying responses after the controller executes. The signature is:

@FunctionalInterface
public interface Middleware {
    Response handle(Request req, Handler next);
}

An implementation example using Java lambdas:

Middleware myMiddleware = (req, next) -> {
    // 1. Code to execute before the handler (e.g. check auth)
    // If aborting, return a Response object directly:
    // if (!authenticated) return new Response("Forbidden", 403);
    
    // 2. Call next.handle(req) to proceed
    Response response = next.handle(req);
    
    // 3. Code to execute after the handler (e.g. add headers)
    response.addHeader("X-Processed-By", "MyMiddleware");
    return response;
};
Key Concepts
  • Short-circuiting: A middleware can abort the request flow early (e.g., if authentication fails) by returning a Response object directly without calling next.handle(req).
  • Execution Order: Global middlewares are registered via router.use() and run for all routes. Route-specific middlewares are passed to router.addRoute() and run only for that route. Middlewares execute in the order they are registered (global first, then route-specific).
  • Request/Response Manipulation: Middlewares can modify the Request object before passing it to the handler, or modify the Response object before returning it to the client.

Reference Middlewares

The framework includes several pre-built middlewares in the Legacy subsystem (src/com/customframework/apps/legacy/Middlewares.java) that can be used globally or on specific routes. For security-specific middleware, see Section 7.3: Pre-routing Filters.

  • adminCookieMiddleware: Handles access control for administrative endpoints (e.g., /admin_home, /api/v1/users, /api/v1/settings). It verifies the role and auth_token cookies. Non-admin users attempting to access these endpoints receive a 403 Forbidden (for API routes) or are redirected to the /login page (for HTML routes).
  • logMiddleware: A utility middleware for detailed access logging. It records the HTTP method and path, extracts and logs the User-Agent header, measures request latency, and logs the response status code and content type.
  • globalSanitizerMiddleware: (Optional/Disabled by default) Automatically HTML-escapes all query parameters in the request URL to prevent injection attacks.
  • securityHeadersMiddleware: Hardens response security by injecting standard defense headers:
    • X-XSS-Protection: 1; mode=block: Instructs legacy browsers with built-in XSS filters to block the page if a reflected Cross-Site Scripting (XSS) attack is detected (largely ignored by modern browsers in favor of CSP).
    • X-Content-Type-Options: nosniff: Prevents the browser from MIME-sniffing the response, ensuring it adheres strictly to the declared content-type.
    • X-Frame-Options: DENY: Prevents the page from being embedded in frames or iframes, protecting against clickjacking.
  • referrerPolicyMiddleware: Sets the Referrer-Policy header to strict-origin-when-cross-origin.
  • requestIdMiddleware: (Optional/Disabled by default) Generates a unique UUID for each incoming request, injects it into the request headers (as X-Request-ID), and adds it to the outgoing response headers.

Example

The following example demonstrates how to register global or route-specific middlewares in the application entry point:

Router router = new Router();

// 1. Register a global middleware (using lambda)
router.use((req, next) -> {
    Logger.log("Request: " + req.getMethod() + " " + req.getPath());
    return next.handle(req);
});

// 2. Define a route-specific middleware
Middleware authMiddleware = (req, next) -> {
    if (req.getCookies().get("session_id") == null) {
        return new Response("Login Required", 401);
    }
    return next.handle(req);
};

// Apply route-specific middleware to /profile
router.addRoute("GET", "/profile", Controller::profileHandler, Collections.singletonList(authMiddleware));
router.addRoute("GET", "/home", Controller::homeHandler); // only runs global middlewares

5. Server Operations

5.1. Starting the Server

To start the server, use the Server.runServer function. It takes a port number and a configured Router instance.

import com.customframework.core.Router;
import com.customframework.core.Server;

Router router = new Router();
// Register routes...

Server.runServer(8080, router);

5.2. Error Handling

  • 404 Not Found: Automatically returned if no route matches the request.
  • 500 Internal Server Error: Automatically returned if an exception occurs during handler execution. The response will include the stack trace for debugging.

6. Best Practices & Performance

6.1. Connection Management

For high-throughput services, ensure client connections are managed efficiently. While the SDK utilizes Java's built-in HttpServer which handles basic socket lifecycle, developers should ensure that client libraries use connection pooling where possible to reduce handshake overhead.

6.2. JSON APIs

In the Java SDK, handlers must return a Response object. To serve JSON, serialize your data to a JSON string and return a Response with application/json content type:

public static Response apiHandler(Request request) {
    // Serialize data to a JSON string
    String json = "{\"status\": \"success\", \"data\": [1, 2, 3]}";
    return new Response(json, 200, "application/json");
}

7. Security & Hardening

7.1. Docker Hardening

To ensure the microservices run securely in production, adhere to the following Docker hardening guidelines:

  • Run as Non-Root User: Never run the application as root inside the container. Ensure the Dockerfile switches to a non-privileged user (e.g., nobody or a custom appuser). 
    RUN groupadd -g 10001 appuser && useradd -u 10000 -g appuser appuser
USER appuser
  • Read-Only Filesystem: Mount the container filesystem as read-only to prevent unauthorized modifications at runtime. Use temporary writes in memory (tmpfs) if necessary. 
    docker run --read-only --tmpfs /tmp:rw,noexec,nosuid my-app-image
  • Linux Capabilities: Drop all default kernel capabilities and only add what is strictly necessary (usually none for simple web servers). 
    # docker-compose.yml example
security_opt:
  - no-new-privileges:true
cap_drop:
  - ALL

7.2. Production Guidelines

  • Resource Constraints: Limit CPU and memory usage to prevent Denial of Service (DoS) attacks from exhausting host resources.
  • Network Isolation: Use internal Docker networks to isolate applications. Only expose the necessary ports (e.g., via a reverse proxy like Nginx or an API Gateway).

7.3. Pre-routing Filters

When developing new features or endpoints, security must be a primary concern. If an endpoint deals with insecure, raw, or user-supplied requests, it is recommended to either utilize the existing security middlewares (such as securityHeadersMiddleware or globalSanitizerMiddleware) or create custom security middlewares to validate, sanitize, and harden the request before it reaches the controller handler.

Using middleware for security controls ensures consistent policy enforcement across routes and keeps controller logic focused on business requirements.

7.4. Dependency Vulnerability Auditing

To prevent the introduction of known vulnerabilities through third-party packages, all projects utilizing the SDK should regularly audit their dependencies. It is recommended to integrate automated scanning tools (such as pip-audit or Safety) into the CI/CD pipeline to detect and alert on deprecated or insecure library versions before deployment.

Always pin dependency versions in requirements files to ensure reproducible and secure builds.

7.5. Secure Logging and PII Protection

While detailed access logging (such as that provided by log_middleware) is essential for operational monitoring and security auditing, developers must ensure that sensitive information is never written to log files. This includes Personally Identifiable Information (PII), session tokens, passwords, and cryptographic keys.

Ensure that log sanitization filters are configured to mask or redact sensitive query parameters (e.g., password, token) and headers (e.g., Authorization, Cookie) before logs are persisted or sent to central log aggregation systems.

8.3. Hardening Auth

To transition from mock authentication to production-grade security:

  1. Password Hashing: Never store or compare passwords in plain text. Use strong hashing algorithms like bcrypt or Argon2id with appropriate salt.
  2. JSON Web Tokens (JWT): Replace static tokens with cryptographically signed JWTs.
    • Implement token expiration (exp claim).
    • Use asymmetric signing (RS256) where the auth service signs tokens with a private key, and microservices verify them using the public key.
  3. Token Storage: Recommend clients store tokens in HttpOnly; Secure; SameSite=Strict cookies to mitigate XSS and CSRF risks.

10. Frequently Asked Questions (FAQ)

Q: How do I access JSON data sent in a POST request?

A: The framework automatically parses JSON bodies if the client sends the Content-Type: application/json header. You can access the parsed dictionary directly using the req.json property on the Request object (e.g., username = req.json.get("username")).

Q: Does the router support dynamic path parameters like /api/users/:id?

A: No. To keep the framework lightweight, the router only supports exact string matching for routes. If you need dynamic parameters, you must register a static prefix route (e.g., /api/users/) and parse the dynamic segments manually by splitting the req.path string in your handler.

Q: Can my controller handler return something other than a Response object (like a Map or String)?

A: No. In the Java SDK, all controller handlers must implement the Handler interface and return a Response object directly. If you want to return JSON, you should serialize your data to a JSON string and construct a Response object with the application/json content type.

Q: Can I register multiple handlers for the exact same route (method + path)?

A: No. The routing table maps the tuple (method, path) to a single handler. If you call router.add_route() multiple times for the same method and path, the last registered handler will overwrite any previous registrations.

Q: In what order do middlewares execute if I use both global and route-specific ones?

A: Middlewares execute in the order they are registered. Global middlewares (registered via router.use()) run first, from the outer-most to the inner-most. After all global middlewares have processed the request, the route-specific middlewares (passed to router.add_route()) are executed in the order they appear in the list, finally passing the request to the controller handler.

Q: How can a middleware abort a request early (e.g., if authentication fails)?

A: To abort a request, the middleware should simply construct and return a Response object directly (e.g., return Response("Unauthorized", status_code=401)) instead of calling and returning next_handler(request). This breaks the chain, preventing any subsequent middlewares or handlers from running.
